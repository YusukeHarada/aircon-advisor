# エアコン設定温度アプリ 要件定義書

## 1. 概要・目的

その日の外気温・湿度に応じた推奨エアコン設定温度と運転モードを提示するアプリ。

最適化の優先順位を次のとおり定義する。

1. 快適性（主目的）：室内の体感を快適域に置く設定温度を算出する
2. 健康（制約）：室内外の温度差の抑制、高湿度時の熱中症予防を制約として快適性の算出値を補正する
3. 省エネ（補足）：主推奨を上書きせず、別案として併記する

## 2. スコープ

### MVP範囲
- 初回オンボーディング（地点設定、個人オフセット初期値の説明）
- 現在地または設定地点の外気温・湿度の取得
- 推奨設定温度と運転モードの算出・表示
- 健康制約による補正
- 省エネ別案の併記
- 寒がり／暑がりの個人オフセット設定
- 「今暑い／寒い」フィードバック入力による簡易オフセット調整
- ユーザー認証（匿名認証によるuid発行）

### 対象外（将来フェーズ）
- スマート家電連携による自動制御（Nature Remo、SwitchBot、メーカークラウド等）
- 室温・湿度センサの実測フィードバック（実測を用いた閉ループ制御）
- 家族・複数ユーザー共有

## 3. 前提・制約

- アプリの役割は「推奨設定温度の表示のみ」。実際の操作はユーザーが行う（フィードフォワード制御）。
- 実際の室温・湿度は取得しない前提。室内環境の個体差はフィードバック入力による個人オフセットで補う。
- 技術構成は Next.js + Firebase（既存構成を踏襲）。
- 実測室温・湿度センサは持たない前提のため、DIなどの内部指標名はユーザー向け画面には出さず、平易な体感表現（「蒸し暑さに注意」等）に変換して表示する。

## 4. 機能要件

| ID | 機能 | 内容 |
|----|------|------|
| F-00 | 認証・オンボーディング | 匿名認証でuidを発行し、初回起動時に地点設定とオフセット初期値（0）を案内する |
| F-01 | 気象データ取得 | 地点の外気温・湿度を取得しキャッシュする |
| F-02 | 推奨設定温度算出 | 外気温・湿度から推奨設定温度を算出する |
| F-03 | 運転モード判定 | 冷房／暖房／除湿／送風／不要を判定し、判定理由を平易な文言で提示する |
| F-04 | 健康補正 | 室内外温度差の上限適用、高湿度時の除湿優先 |
| F-05 | 省エネ別案 | 主推奨に対する省エネ寄りの別案を、主推奨と視覚的に区別して併記する |
| F-06 | 個人オフセット | 寒がり／暑がりのオフセットを設定・反映する |
| F-07 | フィードバック | 「暑い／寒い」入力でオフセットを微調整し、反映結果（現在のオフセット値）と直近入力の取り消し手段を提示する |

## 5. ロジック仕様

### 5.1 不快指数（DI）

体感の基準として不快指数を用いる。

```
DI = 0.81 * T + 0.01 * H * (0.99 * T - 14.3) + 46.3
```

- T：気温（℃）
- H：相対湿度（%）

快適域の目安は DI 70〜72、目標室内DIの初期値は 71 とする。実運用でのフィードバック（「暑い」「寒い」の傾向）を見ながらチューニングする。

### 5.2 冷房時の設定温度算出

目標室内 DI と想定室内湿度から、設定温度 T を逆算する。5.1 の式を T について解くと次のとおり。

```
T = (DI_target - 46.3 + 0.143 * H_in) / (0.81 + 0.0099 * H_in)
```

- DI_target：目標室内不快指数（初期値 71）
- H_in：想定室内湿度（初期値 55%。冷房により外気湿度より低下する想定）

### 5.3 暖房時の設定温度算出

暖房時は湿度の寄与が小さいため、目標室温ベースで算出する。

- 基準室温：20〜22℃
- 個人オフセットを加算して最終値とする

### 5.4 運転モード判定

外気温の閾値でモードを切り分ける。閾値は初期値であり調整可能とする。

| 外気温 | モード |
|--------|--------|
| 25℃以上 | 冷房 |
| 18〜24℃（中間期） | 除湿／送風／不要（湿度で分岐） |
| 18℃未満 | 暖房 |

中間期は湿度が高ければ除湿、低ければ送風または「エアコン不要」を提示する。

### 5.5 健康補正（制約）

快適性の算出値に対し、健康ロジックを制約として適用する。

- 室内外温度差の上限：冷房時、設定温度の下限を `外気温 - 7℃` でクリップする（寒暖差による負荷の抑制）。快適性の算出値がこの範囲内なら補正しない。
- 高湿度時の除湿優先：外気湿度が 70% 以上のとき、除湿運転を優先提示する（熱中症予防）。

快適性を主目的とするため、健康補正は快適域を逸脱する場合のみ作用するガードレールとして機能する。

### 5.6 省エネ別案（補足）

主推奨を上書きせず、別案として併記する。

- 夏：主推奨 +1℃
- 冬：主推奨 -1℃
- 参考値として環境省の目安（夏28℃／冬20℃）を表示可能とする

### 5.7 個人オフセット

最終設定温度に個人オフセット（±1〜2℃）を加算する。フィードバック入力（「暑い」「寒い」）の累積でオフセットを簡易調整する。

### 5.8 実装配置とテスト方針

DI計算（5.1）、設定温度算出（5.2/5.3）、運転モード判定（5.4）、健康補正（5.5）、オフセット反映（5.7）は、入力（外気温・湿度・設定値）から出力（推奨温度・モード）を返す純粋関数として切り出す。クライアント・サーバーいずれからも同一ロジックを呼び出せるようにし、各関数を単体テスト対象とする。

## 6. データモデル（Firestore）

### users/{uid}/settings
- location：地点（気象庁アメダスの観測所番号。5桁の数値コード。例 `44132`＝東京。オンボーディングで現在地から最寄りの観測所番号に変換して保存する）
- offset：個人オフセット（℃、初期値 0）
- comfort_di_target：目標室内DI（初期値 71）
- indoor_humidity：想定室内湿度（初期値 55）

### weather_cache/{location_key}/{timestamp}
- location_key：気象庁アメダスの観測所番号（例 `44132`＝東京）。同一観測所番号のユーザーはキャッシュを共有し、呼び出し回数を抑える。
- temp：外気温（℃）
- humidity：相対湿度（%）
- fetched_at：取得時刻

### users/{uid}/feedback/{id}
- created_at：日時
- signal：hot / cold
- temp_at_feedback：入力時の外気温
- deleted：取り消しフラグ（初期値 false。直近入力の取り消し操作で true にし、オフセット再計算時は除外する）

## 7. 外部データ（気象API）

気象データソースは**気象庁**（アメダス）を採用する（無料・APIキー不要・日本国内の信頼性が高いため）。非公式エンドポイントのため、抽象化レイヤ（インタフェース）を挟んでソースを差し替え可能にしておき、エンドポイント仕様変更や将来のソース切り替えに備える。

API呼び出しと `weather_cache` への書き込みは Next.js のサーバー側（API Routes/Server Actions）に限定する。気象庁APIはAPIキー不要だが、サーバー経由に統一することでレスポンス整形・キャッシュ書き込みを一元化し、将来キー必須なソースへ切り替える場合も変更範囲をサーバー側に閉じられる。

地点は気象庁アメダスの観測所番号（5桁）単位で扱う（6.節参照）。オンボーディング時に現在地（緯度経度）を取得し、最寄りの観測所番号へマッピングして `users/{uid}/settings.location` に保存する。地点候補の一覧は気象庁が公開する観測所一覧（`amedastable.json`）から取得できる。MVPでは主要都市（東京・大阪・札幌・名古屋・福岡・つくば）の観測所番号を暫定的にハードコードしており、全国対応は将来の拡張とする。

呼び出し制限対策として取得結果を Firestore にキャッシュし、1日4回（6時間おき）更新する。

取得失敗時のフォールバック順序：
1. 直近の `weather_cache` を使用し、鮮度（取得時刻）を画面に明示する
2. キャッシュも存在しない場合は固定デフォルト値（季節に応じた仮の気温・湿度）を使用し、推定値である旨を表示する
3. いずれも不可の場合はエラー表示とする

必要データ：外気温、相対湿度。将来的に体感補正へ拡張する場合は風速も取得する。

### 7.1 エンドポイント仕様（検証済み）

- 最新観測時刻：`GET https://www.jma.go.jp/bosai/amedas/data/latest_time.txt` → ISO8601形式の時刻文字列（JST、例 `2026-07-19T12:10:00+09:00`）を返す
- 観測データ（全国一括）：`GET https://www.jma.go.jp/bosai/amedas/data/map/{YYYYMMDDHHMM00}.json`（分単位、秒は常に `00`）。観測所番号をキーに、`temp`（気温）・`humidity`（湿度）等が `[値, 品質コード]` の配列で入る
- 観測所一覧：`GET https://www.jma.go.jp/bosai/amedas/const/amedastable.json`

これらの仕様はWeb検索により公開情報で裏取り済み。本番環境（https://aircon-advisor.vercel.app）にデプロイし、実データ取得も確認済み。

## 8. 画面構成

- オンボーディング（初回起動時）：地点設定（現在地取得または手動選択）、個人オフセットの初期値（0）と役割の簡単な説明
- メイン：今日の推奨（設定温度・運転モード・省エネ別案）、外気温・湿度の表示、データ取得時刻（鮮度）表示
  - 「この推奨温度を目安に手動でエアコンを設定してください」等、ユーザーが取るべき行動を明示する文言を表示する
  - 運転モードの判定理由（例：「湿度が高いため除湿がおすすめです」）を平易な一言で添える
  - 省エネ別案は主推奨とは別の枠・弱いトーンで表示し、視覚的に区別する
- 設定：地点、寒がり／暑がりオフセット、目標DI等のパラメータ（各項目に平易な説明を添える）
- フィードバック：「今暑い」「今寒い」ボタン。入力後は現在のオフセット値を画面に表示し、直近入力を取り消す導線を設ける

## 9. 非機能要件

- オフライン時はキャッシュした直近データで表示し、鮮度（取得時刻）を明示する
- パラメータ（DI目標値、閾値、マスタ）は Firestore の設定コレクション（例：`config/parameters`）で管理し、コード変更・再デプロイなしにチューニングできるようにする
- Firestoreセキュリティルール：`users/{uid}/*` は本人（認証済みuid一致）のみ読み書き可、`weather_cache/*` は認証済みユーザーであれば読み取り可、書き込みはサーバー側（Admin SDK経由）のみに限定する
- 通知・プッシュリマインドはMVP対象外とする。ユーザーが能動的にアプリを開いた際にのみ推奨を提示する
- Safari対応：FirestoreのリアルタイムListen/Write接続がSafariの `http(s)://` 環境で `due to access control checks` エラーにより失敗することがある（Firebase JS SDKの既知の問題）。`initializeFirestore` に `experimentalForceLongPolling: true` を指定して回避する（`lib/firebase/client.ts`）
